# ArborGraph 不足之处分析与改进方向

## 1. 知识图谱构建层面

### 1.1 实体消歧能力不足
**现状**：当前实体合并仅基于名称规范化（大小写统一）和简单字符串匹配。
**问题**：同名但不同含义的实体会被错误合并（如"苹果"指水果 vs 公司），而不同名称但指向同一实体的表述不会被合并（如"AI"和"人工智能"）。
**改进方向**：引入基于语义嵌入的实体消歧，参考 KGGen 的聚类消歧方法。

### 1.2 关系类型粗糙
**现状**：LLM 抽取的关系缺少标准化的类型分类，关系描述是自由文本。
**问题**：难以进行基于关系类型的精确查询和分区，影响多跳推理的质量。
**改进方向**：定义关系类型本体（is-a、part-of、cause-effect 等），在抽取 Prompt 中明确要求标注关系类型。

### 1.3 知识图谱冲突未处理
**现状**：多文本块可能对同一实体产生矛盾的描述或关系，但系统直接合并不做冲突检测。
**问题**：矛盾信息进入知识图谱会导致生成的 QA 对包含矛盾事实。
**改进方向**：参考 Graphusion 的冲突消解策略，在合并前检测并解决矛盾关系。

---

## 2. 数据生成质量层面

### 2.1 缺乏系统化的质量精炼
**现状**：生成的 QA 对经 Critic 验证后直接输出，没有迭代优化环节。
**问题**：一次性生成的质量上限受 LLM 能力限制，部分通过验证的样本仍存在可改进空间。
**改进方向**：参考 Condor 的自反思精炼机制，添加 QA 生成 → 评价 → 修改 → 再评价的迭代循环。

### 2.2 难度控制能力弱
**现状**：多种生成模式（原子/聚合/多跳/CoT）隐含了难度梯度，但没有显式的难度控制参数。
**问题**：无法按需生成特定难度区间的 QA 对，不利于分阶段训练或课程学习。
**改进方向**：参考 Evol-Instruct 的复杂性进化策略，引入显式难度等级和渐进式复杂化机制。

### 2.3 多样性度量缺失
**现状**：依赖 MTLD（词汇多样性）和去重哈希来保证多样性。
**问题**：词汇多样性不等同于语义多样性，可能存在大量语义相近但措辞不同的 QA 对。
**改进方向**：引入语义嵌入空间中的多样性度量（如 embedding 距离分布、聚类熵），参考 NovelSum 的多样性指标。

### 2.4 缺少多轮对话生成
**现状**：所有生成模式都产出单轮 QA 对。
**问题**：真实 SFT 场景中多轮对话数据同样重要，单轮数据不足以训练对话能力。
**改进方向**：参考 UltraChat 的多轮对话生成策略，基于知识图谱子图设计多轮对话链。

## 3. ECE 知识盲点识别层面

### 3.1 Trainee 模型依赖
**现状**：ECE 机制需要部署一个待训练模型（Trainee）来判断命题。
**问题**：增加了使用门槛和 API 成本，很多用户可能没有双模型预算。
**改进方向**：提供无需 Trainee 模型的轻量级知识盲点估计方法，如基于问题难度启发式或知识图谱拓扑特征。

### 3.2 ECE 分数校准精度
**现状**：使用单次判断的 loss 值估计知识盲点程度。
**问题**：单次判断的噪声较大，可能导致错误的优先级排序。
**改进方向**：多次采样取平均，或结合信息增益等其他度量。

## 4. 工程实现层面

### 4.1 缺少异步并发控制
**现状**：知识图谱构建和 QA 生成的 LLM 调用依赖于 RPM/TPM 限制器，但没有全局的并发任务调度。
**问题**：多任务并行时可能超出 API 限额，或资源利用不充分。
**改进方向**：实现全局信号量和优先级队列，支持跨任务的 API 配额共享。

### 4.2 存储层扩展性受限
**现状**：知识图谱和 QA 数据使用 JSON 文件 + NetworkX 内存图存储。
**问题**：大规模数据（超过 100MB 文本）时内存占用过大，且无法支持多进程共享。
**改进方向**：引入 Neo4j、ArangoDB 等图数据库，或 SQLite/PostgreSQL 作为持久化存储。

### 4.3 错误恢复能力不足
**现状**：长时间运行的任务中途失败时，已完成的知识图谱构建结果可能丢失。
**问题**：大文件处理需要数小时，中途失败需要完全重新开始。
**改进方向**：实现细粒度的检查点机制，支持从最后成功的步骤恢复。

### 4.4 潜在的死锁/卡死风险
**现状**：`async_to_sync_method` 装饰器在同步上下文中运行异步代码，结合批量请求管理器的 Future 等待机制。
**问题**：在特定条件下（如 API 持续超时、队列未被消费）可能导致程序卡死。
**改进方向**：为所有 Future 添加超时机制，完善错误传播链路，增加死锁检测。

---

## 5. 评测与验证层面

### 5.1 缺少标准化评测基准
**现状**：评测主要依赖 MTLD、UniEval 等通用指标和人工审核。
**问题**：缺少针对 SFT 数据质量的标准化评测框架，难以横向对比。
**改进方向**：构建 SFT 数据质量评测基准，包含事实准确性、知识覆盖度、多样性、有用性等维度。

### 5.2 缺少自动化回归测试
**现状**：测试文件存在但覆盖不完整，无 CI/CD 集成。
**问题**：代码修改后难以快速验证功能正确性。
**改进方向**：建立 CI/CD 流水线，至少覆盖核心管线的端到端冒烟测试。

---

## 6. 用户体验层面

### 6.1 配置复杂度高
**现状**：运行需要配置多个环境变量、API Key、模型名称、分区参数等。
**问题**：新用户上手门槛高，容易配错。
**改进方向**：提供交互式配置向导，预设常见场景的模板配置。

### 6.2 缺少实时进度展示
**现状**：后端通过轮询更新任务状态，前端定时刷新。
**问题**：长任务处理时用户无法感知进度和中间结果。
**改进方向**：引入 WebSocket 推送实时进度，展示当前步骤和已生成的 QA 对数量。
