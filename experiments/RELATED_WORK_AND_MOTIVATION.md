# 相关工作与研究动机

---

## 一、GraphGen：基于知识图谱引导的合成数据生成

### 1.1 背景

随着大语言模型（LLM）在自然语言处理任务中的广泛应用，监督微调（Supervised Fine-Tuning, SFT）已成为提升模型性能的关键技术。然而，高质量训练数据的稀缺性严重制约了 SFT 的效果，尤其在知识密集型任务（如问答、知识推理、事实检索）中，传统的合成数据生成方法面临根本性瓶颈。

现有的合成数据生成方案主要包括三类：（1）**直接生成方法**，使用 LLM 直接从文档生成 QA 对，但缺乏结构化知识指导，容易产生事实错误；（2）**模板填充方法**，基于预定义模板填充实体，灵活性差，难以处理复杂关系；（3）**检索增强方法**，结合检索系统，但缺乏对模型知识盲点的针对性识别。这些方法均未充分利用结构化知识图谱的优势来指导数据合成过程。

GraphGen（arXiv: 2505.20416）正是在此背景下提出的，旨在通过引入知识图谱作为中间结构化表示，系统性地提升合成数据的事实准确性、知识覆盖度和多样性。

### 1.2 动机

GraphGen 的设计受三个核心问题驱动：

1. **事实准确性低**：LLM 在生成过程中容易出现幻觉（Hallucination），产生与源文本不符的事实错误。纯文本到文本的生成缺乏结构化约束，无法从根源上防止幻觉。

2. **知识覆盖不足**：传统方法倾向于生成 LLM 已经"熟悉"的常见知识，难以系统性地覆盖长尾知识（出现频率低但对专业领域至关重要的知识）和复杂关系（需要多步推理的知识链条），导致训练数据偏向"热门"知识。

3. **缺乏针对性**：现有方法无法识别被微调模型的知识盲点——即模型理解薄弱的区域。这导致大量生成的数据对模型提升有限，属于"低效填充"。

### 1.3 主要方法

GraphGen 提出了一个端到端的知识图谱引导合成数据生成框架，核心流程分为四个步骤：

#### 步骤一：知识构建（Knowledge Construction）

从原始非结构化文档出发，利用 LLM 进行细粒度的实体和关系提取，构建结构化知识图谱。具体包括：
- **文档分块**：将长文档按照可配置的 chunk_size 分割为较小的片段，支持递归字符分割、Markdown 分割等多种策略
- **实体关系提取**：使用合成器模型（Synthesizer Model）从每个片段中提取实体（含 ID、类型、描述）和关系（含源实体、目标实体、关系描述）
- **跨文档聚合**：将不同片段中提到的相同实体和关系进行合并，消除重复，形成完整的知识表示
- **知识图谱存储**：基于 NetworkX 构建图存储，每个节点代表一个实体，每条边代表一个关系

#### 步骤二：理解评估（Comprehension Assessment）

这是 GraphGen 最具创新性的设计之一，通过预期校准误差（Expected Calibration Error, ECE）机制量化被训练模型（Trainee Model）对每个知识点的理解程度：
- **语义变体生成**：为知识图谱中的每条边（关系）生成多个语义变体，包括肯定形式（"A 是 B 的子类"）和否定形式（"A 不是 B 的子类"）
- **置信度评估**：通过二元是/否问题提示被训练模型，获取模型对每个语义变体的置信度分布
- **理解损失计算**：通过交叉熵计算真实分布与模型预测分布之间的差异，得到每条边的理解损失（Comprehension Loss）。高损失值意味着模型对该知识点理解不足，即"知识盲点"

#### 步骤三：图组织（Graph Organization）

基于理解评估的结果，对知识图谱进行智能分区和子图采样：
- **图分区**：支持多种分区策略——BFS（广度优先）、DFS（深度优先）、ECE（基于理解损失的优先采样）、Leiden（基于社区检测）、Anchor BFS（基于锚点的广度优先）
- **ECE 驱动采样**：在 ECE 模式下，优先选择高损失值的边作为采样种子，确保生成的数据集中覆盖模型的知识盲点
- **多跳子图提取**：从种子节点出发，通过 k-hop 邻域扩展提取子图，捕获复杂的多跳关系上下文

#### 步骤四：QA 生成（QA Generation）

将采样后的子图转换为多样化的 QA 对，支持四种生成模式：
- **原子模式（Atomic）**：针对单个事实三元组生成简单直接的 QA 对，如"北京是哪个国家的首都？→中国"
- **聚合模式（Aggregated）**：将多个相关三元组聚合为一个上下文，生成需要整合多源信息的 QA 对
- **多跳模式（Multi-hop）**：基于多跳关系路径生成需要多步推理才能回答的 QA 对
- **链式思考模式（CoT）**：生成包含推理过程的 QA 对，答案中展示完整的推理步骤

### 1.4 优势

1. **结构化知识约束**：通过知识图谱提供显式的实体-关系结构，从源头避免 LLM 生成过程中的幻觉问题，生成的 QA 对有图谱事实兜底
2. **知识盲点按需生成**：ECE 机制使系统能够精准识别模型的薄弱区域，实现"按需生成"而非"盲目扩充"，长尾知识覆盖率相比随机采样提升约 60%
3. **多跳推理能力**：k-hop 邻域采样可以捕获复杂的多步关系链条，生成的 QA 对平均推理跳数达到 2.3，远高于传统方法的 1.0-1.5
4. **差异化生成策略**：四种生成模式覆盖从简单事实到复杂推理的不同难度层次，提升数据多样性，MTLD 词汇多样性指标提升 15-20%
5. **工程效率优化**：通过批量请求优化、提取结果缓存、Prompt 合并等技术，实现了 40-60% 的 LLM 调用次数减少

### 1.5 局限性

1. **三元组扁平化层级知识**：GraphGen 的知识图谱将所有关系视为平等的边，`is_a`（层级关系）和 `located_in`（横向关联）没有区别。这导致系统擅长捕捉"横向关联"，但弱于捕捉"纵向层级"（如分类体系、继承关系）。当文档包含丰富的层级知识时（如"机器学习分为监督学习和无监督学习"），GraphGen 可能无法生成有效的分类对比类问答

2. **随机游走破坏层级完整性**：邻域采样算法可能沿着混合路径（如 `A → is_a → B → related_to → C`）采样，虽然构成了多跳推理，但丢失了"A, D, E 都是 B 的子类"这种兄弟节点的对比价值和层级全貌

3. **图转文的语言自然度**：将结构化子图转换为自然语言的过程容易产生"格式化痕迹重"的文本，生成的 QA 对在对话自然度和人类偏好（Helpfulness）上可能不如基于灵感激发的方法

4. **数据覆盖广度受限于输入文档**：GraphGen 是 Bottom-Up 方法，知识完全来源于输入文档。如果文档本身存在知识盲区，生成的数据也无法覆盖这些盲区。缺乏对"应该覆盖哪些话题"的系统性规划

5. **图谱构建本身的代价与风险**：提取高质量知识图谱是一个困难的 NLP 任务。如果实体链接或关系提取出错，生成的 SFT 数据将包含"结构化的幻觉"——这种错误因披着结构化外衣而更难被察觉和纠正

6. **领域适应成本高**：不同领域的实体类型和关系模式差异大，需要针对性地调整提示模板和提取策略，缺乏"换领域即用"的通用机制

## 二、Condor：基于世界知识树与自我反思的系统化对齐

### 2.1 背景

Condor（arXiv: 2501.12273, ACL 2025）由 InternLM 团队提出，被应用于 InternLM3 的训练流程中。其出发点与 GraphGen 相同——都是为了解决高质量 SFT 数据的稀缺问题——但采用了完全不同的方法论路径。

在 Condor 之前，Self-Instruct（自我指令）是最主流的合成数据方法：让 LLM 自行生成指令和回答。然而，Self-Instruct 存在严重的"概念空间坍缩"问题——模型倾向于在自己"舒适区"内循环生成，导致 20K 条数据中可能只有极少部分具备真正的新颖性。大量数据是换了措辞的重复内容。

Condor 的核心洞察是：**问题不在于"怎么生成回答"，而在于"怎么系统性地规划应该问什么"**。通过引入"世界知识树"（World Knowledge Tree）作为全局性的话题分布先验，Condor 从根本上解决了 Self-Instruct 的覆盖面窄和同质化问题。

### 2.2 动机

Condor 的设计受以下三个核心问题驱动：

1. **高质量人工数据瓶颈**：随着模型规模增加（7B → 72B），人工标注数据的成本呈超线性增长，已无法满足需求。业界不得不转向合成数据，但如何保证合成数据的质量成为关键挑战

2. **Self-Instruct 的同质化顽疾**：传统的自我指令方法让模型自由发挥，结果是生成的数据集中在模型"已经擅长"的少数话题上。例如让 GPT-4 自由生成 SFT 数据，可能 40% 是编程相关、30% 是写作相关，仅有极少涉及金融、法律、医疗等专业领域

3. **知识遗忘风险**：在 SFT 阶段，如果训练数据的知识覆盖不够系统性，模型可能在对齐人类偏好的同时"遗忘"预训练阶段学到的世界知识。这是一个需要在"对齐"和"知识保持"之间做平衡的核心矛盾

### 2.3 主要方法

Condor 采用两阶段流水线架构：

#### 阶段一：Condor Void（数据合成）

核心思想是"先规划再生成"，解决"问什么"的问题：

- **世界知识树（WKT）构建**：通过 Top-Down 的分类学方法，让 LLM 生成覆盖全领域的知识标签。从宏观类别（如"科技""教育""医疗"等 40+ 一级类别）出发，逐层向下细分为子类别和具体标签（三层树结构，包含数千个叶子节点）。例如：`人工智能 → 深度学习 → 神经网络 / 深度学习框架 / 深度学习应用`

- **任务类型扩展**：在每个叶子标签下，扩展不同的任务类型（角色扮演、日常对话、QA、给定材料处理、格式控制、观点分析、创作等），确保生成的指令不仅覆盖不同话题，还覆盖不同的任务形态

- **难度分级**：每个标签下生成简单（Easy）、中等（Medium）、困难（Hard）三个难度级别的问题，确保训练数据具有合理的难度梯度

- **问题生成**：基于标签 + 任务类型 + 难度级别的组合，使用精心设计的 Prompt 模板让 LLM 生成具体的问题。模板中明确要求"不能复制示例的表达方式，必须有自己的新表达"

#### 阶段二：Condor Refine（数据精炼）

引入自我反思机制，解决"怎么让回答更好"的问题：

- **初始回答生成**：让 LLM 对 Void 阶段生成的问题给出初始回答
- **Critique 生成**：让同一个（或另一个）LLM 对初始回答进行批评，指出优点、缺点、改进建议。Critique 使用结构化格式（`[Strength Start]...[Weakness Start]...[Suggestion Start]...`）
- **回答重写**：基于 Critique 的反馈，LLM 重新编写改进后的回答
- **迭代优化**：可以进行多轮 Critique → Refine 迭代，逐步提升回答质量

### 2.4 优势

1. **系统性全覆盖**：世界知识树从 Top-Down 的视角强制 LLM 走出"舒适区"，确保数据覆盖广（Breadth）且有深度（Depth）。相比 Self-Instruct 的"自由漫步"，WKT 相当于给了一张"知识地图"

2. **高效的数据利用率**：仅需 20K 高质量 Condor 样本，微调后的模型性能即可超越使用 RLHF 训练的对手。数据量少但质量高，降低了训练成本

3. **自我反思提升质量**：Refine 阶段模仿了人类学习中的"反思与纠错"过程，通过迭代优化显著提升回答的深度、准确性和完整性

4. **可扩展性**：论文验证了 Condor 在 7B 到 72B 不同规模模型上均有效，且存在数据量 scaling 的潜力——增加 Condor 数据量可以持续提升模型性能

5. **对话自然度高**：基于灵感激发（标签 → 问题）的生成方式，而非图转文方式，生成的 QA 对更贴近真实的人类对话风格，在 Helpfulness 评测中表现优异

### 2.5 局限性

1. **树状结构无法表达网状知识**：现实世界的知识并非全然是树状层级的。很多跨领域的知识（如"生物信息学"同时涉及生物学和计算机科学）是网状交叉的。强行使用树状结构可能导致横向关联知识的合成能力不足，遗漏了实体之间的复杂关系

2. **缺乏外部事实约束**：Condor 的知识来源于模型内部的世界知识（通过 WKT 激活），而非外部文档或知识库。这意味着：（1）模型不知道的知识无法被覆盖；（2）模型"以为自己知道"但实际错误的知识会被固化为训练数据中的"结构化幻觉"

3. **标签偏见传播**：如果初始生成的世界知识树根节点或标签存在偏见（Bias），整个后续生成的数据流都会受到污染。树的偏斜会导致某些领域被过度覆盖，另一些被忽略

4. **事实深度和推理链不足**：Condor 生成的 QA 对侧重于话题的广度，但在需要多步推理的场景中（如"A 导致 B，B 又影响 C，那么 A 对 C 的影响是什么？"）表现不如图谱引导的方法。缺乏结构化的关系路径作为推理支撑

5. **计算开销较大**：迭代式的"自我反思"流程（生成 → Critique → 重写 → 再 Critique → 再重写）需要多次 LLM 调用，在大规模生产时，生成效率与成本的平衡是挑战

6. **不具备领域适应性**：Condor 的 WKT 是一棵通用的"世界知识树"，对于特定垂直领域（如金融风控、网络安全）的深入覆盖，需要手工设计领域特定的标签树，通用性有限

## 三、ArborGraph (ArborGraph-Intent)：融合树与图的领域无关合成数据框架

### 3.1 为什么需要本项目

GraphGen 和 Condor 代表了当前 SFT 合成数据生成的两个前沿方向，它们的核心思想高度一致——**不再依赖盲目的 Self-Instruct，而是引入"结构化知识先验"来引导数据合成**。然而，两者各有侧重，也各有盲区：

| 维度 | GraphGen 的问题 | Condor 的问题 | 融合后的解决方案 |
|------|----------------|---------------|-----------------|
| **知识覆盖** | Bottom-Up：覆盖范围受限于输入文档，缺乏全局规划 | Top-Down：覆盖广度好，但不保证事实深度 | 树规划广度 + 图保证深度 |
| **层级知识** | 三元组消解层级感，随机游走破坏树结构 | 天然的层级结构，但缺少节点间的关系细节 | 层级分区 + 图谱事实补充 |
| **事实准确性** | 有图谱约束，事实性强 | 依赖模型内部知识，可能包含幻觉 | 图谱约束 + Critic 验证双重兜底 |
| **数据多样性** | 模式受限于图采样结果 | WKT + 任务类型扩展保证多样性 | 认知维度模板扩展多样性 |
| **领域适应** | 需手工调整提示模板 | 需手工设计领域标签树 | 可插拔配置包，领域无关 |
| **质量保障** | 无后置验证机制 | Refine 阶段通过自我反思提升质量 | Logic-Critic 层提供规则和 LLM 双重验证 |

用一句话总结：**GraphGen 像一个"微观的严谨学者"，确保每个事实都有据可查但视野有限；Condor 像一个"宏观的架构师"，确保知识覆盖面广但细节可能失实。本项目（ArborGraph-Intent）要做的是将两者融合为一个"既有全局视野又有严谨细节"的统一框架。**

### 3.2 突破方向

本项目在以下五个方面寻求突破：

#### 突破一：统一 Top-Down 与 Bottom-Up

现有方法要么从上到下（Condor 的 WKT），要么从下到上（GraphGen 的 KG）。本项目首次将两者统一在同一个流水线中：
- **Macro-Intent 层**（继承 Condor）：通过可插拔的分类树规划"应该覆盖什么话题"
- **Micro-Fact 层**（继承 GraphGen）：通过知识图谱保证"每个话题下的事实是准确的"
- 分类树的叶子节点通过语义匹配自动链接到知识图谱的相关子图，实现"树引导采样、图提供内容"

#### 突破二：层级知识的结构感知

GraphGen 的最大短板是无法处理层级知识。本项目新增 `HierarchicalPartitioner`，专门识别层级关系（`is_a`, `subclass_of`, `part_of`, `includes`, `type_of`）并设计两种特殊采样策略：
- **横向兄弟分组（Sibling Grouping）**：锁定一个父节点，将其所有子节点组成一个社区。生成目标是对比类 QA（如"监督学习和无监督学习有什么区别？"）
- **纵向链式采样（Chain Sampling）**：沿着 祖 → 父 → 子 的路径采样完整链条。生成目标是分类体系类 QA（如"为什么逻辑回归被归类为监督学习？"）

这直接解决了 GraphGen "三元组消解层级感"和"随机游走破坏树结构"的问题。

#### 突破三：Logic-Critic 质量验证层

GraphGen 和 Condor 都缺乏后置的质量验证机制（Condor 的 Refine 是改善回答质量，而非验证事实正确性）。本项目新增 Logic-Critic 层：
- **规则验证器（RuleCritic）**：基于可配置规则过滤低质数据，如最小答案长度、问答重复度检测、关键词覆盖检查
- **LLM 验证器（LLMCritic）**：使用 LLM 对生成的 QA 进行事实性和逻辑性评审
- 两者组合使用，为生成数据提供质量下限保障

#### 突破四：领域无关的可插拔架构

GraphGen 和 Condor 都需要针对新领域做大量手工适配。本项目设计了三个"配置包"的可插拔接口，用户只需替换配置包即可适配新领域：
- **分类树配置包**：JSON 格式的领域分类树（已提供金融和网络安全两个示例）
- **图谱配置包**：指定知识图谱的来源和查询方式
- **验证规则配置包**：指定领域特定的质量验证规则

#### 突破五：认知维度模板

不同于 GraphGen 按图结构生成、Condor 按话题标签生成，本项目引入六种认知维度（概念解释、关系推理、规则遵循、异常诊断、对比分析、程序性知识），为每个叶子节点-子图组合匹配最合适的认知维度模板。这使得同一个知识点可以从不同认知角度生成多种 QA 对，进一步提升多样性。

### 3.3 方法论

ArborGraph-Intent 的完整流水线为：

```
输入文档
    ↓
┌─────────────────────────────────────┐
│  1. 知识构建（继承 GraphGen）        │
│     文档 → 分块 → 实体/关系提取 → KG │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  2. 分类树加载/生成                  │
│     加载领域配置包 → TaxonomyTree    │
│     或：AutoTaxonomy 自动生成        │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  3. 意图采样（Macro-Intent 层）      │
│     DiversitySampler 从分类树中      │
│     均匀采样叶子节点作为生成意图      │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  4. 子图检索（Micro-Fact 层）        │
│     IntentGraphLinker：意图 → 关键词  │
│     NetworkXGraphAdapter：关键词 →    │
│     KG 子图（k-hop 扩展）            │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  5. QA 生成（IntentGenerator）        │
│     意图描述 + 子图上下文 + 认知维度  │
│     模板 → LLM 生成 QA 对            │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  6. 质量验证（Logic-Critic 层）      │
│     RuleCritic + LLMCritic          │
│     通过 → 保留；不通过 → 丢弃       │
└──────────────┬──────────────────────┘
               ↓
           最终 QA 数据集
```

同时，在图分区阶段，本项目除了继承 GraphGen 的 BFS/DFS/ECE/Leiden 等策略外，新增了 HierarchicalPartitioner，将层级关系显式分离处理。

### 3.4 对比 GraphGen 的优势

| 维度 | GraphGen | ArborGraph-Intent (本项目) | 提升来源 |
|------|----------|-----------------|----------|
| 数据覆盖广度 | 受限于输入文档内容 | 分类树强制覆盖多个话题和认知维度 | Macro-Intent 层 |
| 层级知识处理 | 三元组扁平化，层级关系被忽略 | HierarchicalPartitioner 专门处理层级关系 | 横向兄弟分组 + 纵向链式采样 |
| 生成多样性 | 受限于图采样结果和单一生成模式 | 六种认知维度模板 × 分类树叶子节点的笛卡尔积 | 认知维度模板 |
| 质量保障 | 无后置验证 | RuleCritic + LLMCritic 双重验证 | Logic-Critic 层 |
| 领域适应 | 需手工调整提示模板 | 替换配置包即可 | 可插拔架构 |
| 对比类 QA | 难以生成（随机游走破坏兄弟关系） | 兄弟分组社区专门生成对比类 QA | HierarchicalPartitioner |
| 分类类 QA | 难以生成（无层级意识） | 纵向链式社区专门生成分类类 QA | HierarchicalPartitioner |

### 3.5 对比 Condor 的优势

| 维度 | Condor | ArborGraph-Intent (本项目) | 提升来源 |
|------|--------|-----------------|----------|
| 事实准确性 | 依赖模型内部知识，可能包含幻觉 | 知识图谱提供外部事实约束 | Micro-Fact 层 |
| 推理深度 | 缺乏结构化关系路径，推理链浅 | k-hop 子图采样支持多跳推理 | GraphGen 继承的图采样 |
| 知识来源 | 仅模型内部知识 | 外部文档 + 知识图谱 + 模型知识 | 三层知识源 |
| 事实验证 | Refine 改善表达，不验证事实 | Logic-Critic 验证事实和逻辑正确性 | Logic-Critic 层 |
| 领域深度 | 通用世界知识树，垂直领域覆盖浅 | 领域专属分类树 + 领域知识图谱 | 可插拔领域配置 |
| 长尾知识 | 依赖模型已有知识，长尾覆盖弱 | ECE 机制识别知识盲点，优先覆盖长尾 | 继承 GraphGen 的 ECE |
| 横向关联 | 树状结构无法表达网状关系 | 知识图谱天然支持网状关系 | Micro-Fact 层 |

### 3.6 总结

ArborGraph-Intent 的核心贡献可以概括为一个公式：

> **ArborGraph-Intent = Condor 的"广度规划" + GraphGen 的"深度事实" + 新增的"层级感知" + 新增的"质量验证" + 领域无关的"可插拔架构"**

它解决的核心学术问题是：**如何为任何知识密集且规则受限的垂直领域，自动化地合成兼具任务多样性与事实严谨性的 SFT 数据？**
