# SGT-Gen 技术报告（四）：不足、风险与演进路线

## 1. 当前不足与风险

### 1.1 工程层不足
- 任务状态与产物映射仍依赖文件系统约定，缺少统一元数据索引。
- 部分脚本仍以“实验辅助”形态存在，参数约束与错误提示不够强。
- 端到端可观测性不足（缺少跨阶段 trace id、阶段耗时分桶、失败类别统计）。

### 1.2 算法层不足
- 层级关系识别目前主要依赖关系类型命中，对隐式层级文本鲁棒性有限。
- Critic 目前以规则和单轮 LLM 评估为主，对高难推理错误仍可能漏检。
- 多样性控制以模板+比例为主，对风格和话题漂移缺少自适应调节。

### 1.3 实验层不足
- 现有计划强调可执行，但尚需积累稳定的多轮复现实验记录。
- 跨领域泛化评估（尤其低资源领域）仍需更多真实文档验证。
- 下游增益与数据质量指标之间的因果链仍需更系统的统计分析。

## 2. 风险优先级（建议）

| 优先级 | 风险 | 影响 | 缓解策略 |
|---|---|---|---|
| P0 | 运行卡住/长耗时无反馈 | 任务不可用 | 批处理锁与超时策略、阶段心跳、失败重试上限 |
| P0 | 生成事实偏移 | 数据污染 | Critic 阈值提升、引用证据链、人工抽检闸口 |
| P1 | 层级识别误判 | 特色能力弱化 | 关系词典扩展、弱监督校正、层级一致性校验 |
| P1 | 产出不可追溯 | 实验不可复现 | 统一 run_id 与配置快照、结果哈希 |
| P2 | 前后端术语不一致 | 维护成本提升 | 命名规范、API 字段迁移策略、兼容窗口 |

## 3. 三阶段演进路线

### 阶段 A：稳定性与可观测性
- 增加任务级 trace id 与阶段耗时明细。
- 将关键异常归类（依赖、网络、解析、逻辑）并形成告警模板。
- 增加“最小可复现包”导出（配置+输入清单+日志索引）。

### 阶段 B：质量闭环
- 引入多裁判评估（规则 + LLM + 统计异常）与一致性投票。
- 为高风险样本输出“证据链片段”，便于人工快速审核。
- 对失败样本建立回流池，支持模板与参数自动修订。

### 阶段 C：泛化与成本优化
- 支持领域适配配置包自动推荐（taxonomy/critic/routing）。
- 引入模型路由：高难样本走强模型，低难样本走轻量模型。
- 在保证质量阈值前提下，优化 token 成本与吞吐。

## 4. 关键验收标准（建议）
- 稳定性：典型任务“无卡死”完成率 >= 99%。
- 质量：关键维度（准确性、可用性）评分持续高于基线。
- 效率：在同等质量门槛下，单位样本平均成本逐步下降。
