# SGT-Gen 技术报告（二）：系统架构与核心流程

## 1. 系统边界与职责划分

### 1.1 三层结构
- `frontend/`：Vue 管理界面，负责任务创建、配置管理、结果查看。
- `backend/`：FastAPI 任务编排与服务层，负责任务生命周期管理、日志与输出落盘。
- `graphgen/`：核心算法引擎（本次统一命名为 SGT-Gen 核心引擎），负责从文档到 QA 的主流程。

### 1.2 模块职责
- 文档处理：读取、切分、分块缓存。
- 知识抽取：实体关系抽取并写入图存储。
- 结构组织：图分区与层级结构识别。
- 数据生成：多模式 QA 生成与格式化导出。
- 质量控制：规则批评器与模型批评器。

## 2. 数据流

1) 输入文档进入 `read -> split`。  
2) 文本块进入 KG 构建，产出 `graph_storage`。  
3) 可选执行 `quiz_and_judge`，为边/节点写入 loss。  
4) 分区器生成社区/子图批次。  
5) 生成器产出 QA，写入 `qa_storage`。  
6) 后端将结果保存为任务输出文件，并更新任务状态。

## 3. 关键设计

### 3.1 可组合生成模式
- `atomic`：基础事实问答。
- `聚合`：聚合叙述问答。
- `multi_hop`：多跳推理问答。
- `cot`：带推理链条问答。
- `hierarchical`：层级结构感知问答。

### 3.2 可插拔分区策略
- BFS/DFS：结构遍历型。
- ECE：loss 导向采样型。
- Leiden：社区检测型。
- Hierarchical：层级关系显式建模型（新项目重点）。

### 3.3 可插拔质量控制
- `RuleCritic`：规则约束（长度、格式、重复率、关键词覆盖等）。
- `LLMCritic`：语义/事实层面的模型判定。

## 4. 稳定性改进（本次重构）

### 4.1 批处理管理器修复
- 问题：批次队列在“部分满载 + 定时触发 + 并发添加”下存在残留请求未消费风险，表现为任务长时间无进度。
- 改动：新增 `process_lock`、`_drain_batches(force)` 与 `_process_one_batch(force)`，并在达到 `batch_size` 时主动 drain；保留定时器处理尾批。
- 结果：请求消费路径从“单次触发”改为“持续 drain 到不可处理”，显著降低卡住概率。

### 4.2 前端/后端旧模块清理
- 删除仅 Gradio 使用的遗留文件，保留任务管理与工作目录工具。
- 同步移除 `environment.yml` 中 gradio 依赖，减少无效安装与冲突面。

## 5. 运维与排障建议
- 强制记录每批请求的入队/出队计数与耗时分位数。
- 对长任务设定心跳指标（每 N 秒更新任务状态）。
- 对 `LLM timeout / rate limit` 分别统计并输出到任务摘要。
